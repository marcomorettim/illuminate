# Chapter 8: Defining the Problem

## Core Idea
A problem is a **gap** between the result you have (the **Undesired Result, R1**) and the result you want (the **Desired Result, R2**). Define it with the Problem-Definition Framework, and it converts mechanically into an SCQA introduction — the problem definition *is* the skeleton of your document.

## Frameworks Introduced
- **Problem = gap between R1 and R2.** The solution is how to get from R1 to R2. Stated this way, **a well-defined problem has no "alternative solutions"** — a candidate either closes the gap or it doesn't.
- **Sequential Analysis** (five questions in logical sequence): 1. Is there / likely to be a problem? 2. Where does it lie? *(→ define the problem → introduction)* 3. Why does it exist? *(→ structure the analysis)* 4. What could we do? 5. What should we do? *(→ find the solution)*.
- **The Problem-Definition Framework**: **Situation → R1 → R2**, where the Situation = a **Starting Point / Opening Scene** (an existing structure or process) plus a **Disturbing Event**. Three questions define it:
  - What's going on? (Starting Point + Disturbing Event)
  - What don't we like about it? (R1)
  - What do we want instead? (R2)
- **Disturbing Event** is **External** (new competitor, new technology, policy shift), **Internal** (added a process, new system, new market), or **Recently Recognized** (lagging performance, sub-par results, research implying a shift).
- **Convert to an introduction**: read the framework **left-to-right and down**; the *last thing known by the reader* is always the Complication.

## Key Concepts
- **R1 / R2** — Undesired Result / Desired Result. State R2 **specifically and quantifiably** so you know when you've reached it and can choose among solutions.
- **Starting Point / Opening Scene** — the stable structure or process where the problem originated.
- **Disturbing Event** — what threatens the stable scene and triggers R1.

## Mental Models
- Think **dramatically**: the curtain opens on the Opening Scene; then something happens (the Disturbing Event) that launches the action and threatens R1.
- The reader's Question depends on **where they stand toward a solution** (Minto's seven situations — e.g., don't know how to get R1→R2; think they know but aren't sure; know but can't implement; tried and it failed; several candidates; can't state R2; not sure they're at R1). Identify the case before writing.

## Anti-patterns
- **Not noticing the reader has already acted.** If a solution is already proposed/tried, the Question changes (to "Is it right?" / "How implement?" / "It failed, now what?"). Missing this mis-frames the whole document.
- **Manufacturing a Disturbing Event** you don't actually have — if you can't identify it, skip it and go straight to R1.

## Worked Example
A company has sold industrial real estate for 30 years by a tried-and-true method — salesmen **list prospects, write a script, deliver the message** — growing sales ~10% a year. **Starting Point/Opening Scene**: that selling process, expected to yield **R2** (continued 10% growth). **Disturbing Event**: quarterly projections show sales *down* 10%. **R1**: the threat that growth will fall short. The gap is the problem; its causes must lie in the Opening Scene's activities — the list is stale, and/or the script isn't punchy, and/or the delivery is weak. Read left-to-right-and-down, it becomes the introduction: **S** — "tried-and-true approach for 30 years, +10%/yr"; **C** — "projections now show −10%, threatening year-end figures"; **Q** — "How can we ensure continued growth?"

## Key Takeaways
1. Define every problem as an R1→R2 gap with a specific, measurable R2.
2. The problem definition converts directly into the introduction — no separate drafting.
3. Always locate where the reader already stands toward a solution; it sets the Question.
