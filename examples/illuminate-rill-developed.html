Rill · Second Wind · dormant-user reactivation

The campaign has to earn the re-open, not demand it.

Around 1.4M Rill users have quietly gone dormant — no app opens, auto-save paused, the round-up habit lapsed. Rill's value compounds silently in the background, so a dormant user never feels a sharp moment of churn: there is no cancellation event to react to, only a slow fade. This brief is not an argument about whether to win them back. It is the machine that does it.

The reactivation machine

Dormant users are scored into cohorts nightly, split off a 10% holdout, routed to a cohort track, escalated up a four-rung ladder across channels, and either reactivate — at which point every pending send is cancelled — or fall through to an evergreen Winback. Hover a stage; each is its own page.

Why it is delicate

Reactivation is materially cheaper than paid acquisition of an equivalent active user, and reactivated savers resume depositing, growing assets under management.

But the same silence that makes dormancy easy to ignore makes reactivation delicate. These are people who granted an app permission and, often, a phone number and a bank connection. Over-messaging them, or messaging a generic "we miss you," is the fastest way to convert a dormant-but-retained user into an opt-out or an uninstall.

So the dormant base is treated as a fragile asset: the campaign may spend a small, measured amount of its goodwill to reactivate, and the guardrails cap the spend. The one rule that governs the whole machine —

The one ruleThe moment a reactivation fires, all pending sends are cancelled and the user exits to Success. No "come back" message after they have already come back.

The dormant base is not one audience. It is four.

Every dormant user lands in exactly one of four mutually-exclusive cohorts, scored nightly and re-checked at every step against the live data extension. Each cohort stopped for a different reason, so each needs a different reason to come back — and a different answer to the incentive question. Nightly scoring — the code that assigns a cohort

A nightly warehouse job recomputes each user's dormancy state and cohort from app-open recency, transfer recency, auto-save status, balance, and withdrawal history — then writes it to the SecondWind_Audience data extension keyed on Subscriber ID. Cohorts are mutually exclusive by the priority order below.

dormancy_cohort.sqlSQL · nightly
-- Dormant = no app open 30d+ AND (auto-save off OR no transfer 45d+),
-- excluding closed / opted-out. Cohort assigned by priority (first match wins).
WITH dormant AS (
  SELECT subscriber_id, pot_balance, roundup_total, autosave_status,
         last_open_days, last_transfer_days, ever_funded, withdrew_flag
  FROM user_daily_snapshot
  WHERE last_open_days >= 30
    AND (autosave_status = 'off' OR last_transfer_days >= 45)
    AND account_status NOT IN ('closed', 'withdrawn_full')
    AND messaging_opt_out = FALSE )
SELECT subscriber_id,
  CASE
    WHEN ever_funded = FALSE              THEN 'C1_NeverFunded'
    WHEN withdrew_flag = TRUE              THEN 'C4_WithdrewPartial'
    WHEN roundup_total > 0 AND pot_balance > 0 THEN 'C2_FundedLapsed'
    ELSE                                  'C3_RoundUpDrifted'
  END AS cohort
FROM dormant;

Suppression on success — the highest-risk logic

The single most important rule, and the highest-risk bug at soft launch: the moment a reactivation event fires, all pending sends are cancelled and the user exits to Success. Messaging someone the day after they came back is the classic failure.

suppression_on_success.jsJourney · entry-event
// Fires on any reactivation signal. Re-checked at every wait step too,
// so stale segmentation can never message someone who already came back.
on('reactivation_event', async (user) => {
  const reactivated =
       user.autosave_status === 'on'
    || user.has_recurring_transfer
    || user.qualifying_deposit_in_window;

  if (reactivated) {
    await journey.cancelPendingSends(user.subscriber_id);  // kill queued P2/P3/P4
    await journey.exit(user, 'SUCCESS');              // no "come back" after they came back
    await nurture.enroll(user, 'new_active_saver');
  }
});

Orchestration — every step

  • Channel priority Push › SMS › Email, gated by permission; in-app additive.
  • Frequency cap 1 / 48h, max 4 across the journey (Cohort 4: 3).
  • Quiet hours no push/SMS 21:00–08:00 local; queued to next window.
  • Global governance a security/servicing message from another journey is prioritised; Second Wind defers.

Entry throttle

A nightly Automation writes newly-dormant users into the entry data extension. Entry is throttled so no user enters within 21 days of a prior Second Wind exit — the re-entry lock that protects the asset.

Immediately on entry: the deterministic 10% holdout split, then the cohort decision split into one of four tracks.

Personalization & fallback — the render logic

Cohort drives which variant is selected; every token has a safe default so no message renders a blank or a broken sentence. A message that would render "You've saved £ without noticing" must never send — the send is held and the copy variant switched.

SET @roundup = AttributeValue("RoundUpTotal")
SET @name    = AttributeValue("FirstName")

  SET @title = Concat("Your first round-up is ready, ", IIF(EMPTY(@name),"there",@name), ".")
ELSE
  SET @title = Concat("You've saved £", @roundup, " without noticing.")
ENDIF

Incrementality — the reported number

The 10% control receives nothing; reported lift is targeted-reactivation minus holdout-reactivation. Gross reactivation flatters the campaign by counting organic returns that would have happened anyway.

incremental_reactivation.sqlSQL · read
-- Headline = incremental, not gross. Reactivated within 30d of journey entry.
SELECT cohort,
  AVG(CASE WHEN arm = 'targeted' AND reactivated_30d THEN 1.0 ELSE 0 END) AS tgt_rate,
  AVG(CASE WHEN arm = 'holdout'  AND reactivated_30d THEN 1.0 ELSE 0 END) AS hold_rate,
  AVG(CASE WHEN arm = 'targeted' AND reactivated_30d THEN 1.0 ELSE 0 END)
  - AVG(CASE WHEN arm = 'holdout' AND reactivated_30d THEN 1.0 ELSE 0 END) AS incremental_pp
FROM second_wind_results
GROUP BY cohort;

Retention read 30 / 60 / 90

Beyond the 30-day reactivation, track the reactivated users' active-rate to distinguish a durable reactivation from a one-tap blip. A cohort that reactivates at 10% but re-lapses to 3% active by day 60 is worse than one that reactivates at 7% and holds 6%. Target ≥55% still-active at 60 days.

Experiment matrix (v1)

  • Test A — incentive vs value-only (Day-10, Cohorts 1&2): net of the holdout. Kill it if it only subsidises would-have-returned users.
  • Test B — reach: push-only vs push+SMS. Does the SMS fallback lift Reached→Reactivated enough to justify cost + opt-out risk?
  • Both read against the shared holdout, pre-registered stopping rules, MDE agreed with Data.

The caps matter as much as the target.

The dormant base is an asset; the campaign may spend a small, measured amount of its goodwill — the guardrails cap the spend. A win on the primary KPI does not count if any guardrail breaches, and a breaching arm pauses automatically.

The compliance box — tighter than retail

  • Reach is the binding constraint, not creative: push ~45% of the base, SMS ~30% consented, email the rest — raising cost and opt-out exposure.
  • Consent is a hard gate: reactivating a lapsed user is not a lawful basis to message a number that never consented.
  • Financial-promotion rules: fair, clear, not misleading; no guaranteed returns; capital-at-risk where micro-invest is referenced.
  • The risk is asymmetric: a lost opt-out is harder to win back than a missed reactivation — so restraint is the default.

Held open

What the brief deliberately leaves unresolved.

Four questions the brief names as open — carried honestly rather than answered prematurely.

Q1Should Withdrew-Partial receive any reactivation messaging in v1?›
Or should Cohort 4 be held for a dedicated trust-rebuilding track later? The brief leans toward a softer cadence and no incentive, but whether to message them at all is unsettled.
Q2Is a £5 boost the right incentive magnitude?›
Or does a match-style mechanic — "we'll match your first restart save up to £X" — reactivate more durably? Test A reads incrementality, but the mechanic itself is open.
Q3What is the right handoff to evergreen Winback?›
One touch a month, or quarterly — before a dormant user should simply be left alone? The 30-day journey ends; the long tail's cadence is undecided.
Q4Do involuntary lapses need their own register?›
Should users whose dormancy coincides with a failed payment (an involuntary lapse) get a different creative register from those who chose to pause? The brief flags the distinction but doesn't yet split on it.
