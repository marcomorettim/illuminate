S — Situation

      
A 31-agent survey of the agentic OS literature produced 162 falsifiable claims across 8 structural domains. The field had converged on the shape of the problems. Not on the mechanisms that solved them.

    

    

      
C — Complication

      
7 of those 162 claims were fabrications — specific numbers, plausible topics, false attributions. 53 more contested. The methods that seem most sophisticated — large models for determinism, aggressive summarization, runtime self-modification — are the least reliable.

    

    

      
Q — Question

      
How do you translate verified research findings into a production system that stays honest about what it can and cannot do — given that the field the research surveyed cannot reliably verify its own output?

Open Problems

  
OP1–OP7 — status in the Greenhouse

  
7 open problems remain from the original whitepaper. No system has solved them. The Greenhouse has partial implementations or explicit gaps for each. OP3 is the hardest — no implementation plan exists.


  


    

      
OP1

      
Long-horizon evaluation

      

      
Partial

      
governance.py → LLMJudge + AccountabilityChain

      
LLMJudge provides per-decision evaluation. AccountabilityChain provides cross-session traceability. Long-horizon evaluation (across sessions, over weeks) requires reflection mechanisms not yet implemented — nightly meta-agent synthesizing patterns across semantic facts is a known gap.

    


    

      
OP2

      
Multi-tenant isolation at scale

      

      
Partial

      
sandbox.py + SoilTracker

      
No mainstream in-process framework provides first-class tenant isolation. Letta is marketed as multi-tenant but is single-tenant in self-hosted form (single shared password). The Greenhouse isolates via soil FSM state but not at hardware or OS process level.

    


    

      
OP3

      
Agent identity & trust

      

      

        OPEN
        no implementation plan
      

      
no orky/ file

      
No cross-protocol identity standard exists. The Greenhouse has no agent identity layer. Each session is identified by its soil state, not a persistent cryptographic identity. Ed25519 signatures are named in the architecture but not implemented at the agent-identity layer. This is the most open of the 7 problems.

    


    

      
OP4

      
Cost governance

      

      
Partial

      
rtk_vault.py → tokens_saved

      
tokens_saved is a governance proxy. There is no budget-cap enforcement — the 15× multiplier is tracked but not bounded. Full cost governance would require per-session and per-task budget limits with automatic escalation.

    


    

      
OP5

      
Deterministic replay

      

      
Partial

      
replay.py → ReplayJournal

      
ReplayJournal captures session events with prev_hash tamper-detection. Replay to a specific checkpoint is supported. Full deterministic replay requires temperature=0 enforcement across all model calls — and even then, the 120B+ determinism finding means determinism degrades with model scale.

    


    

      
OP6

      
Safe self-modification

      

      
Partial

      
governance.py → CorrectionRegister

      
CorrectionRegister prevents infinite self-repair loops (max 3 attempts, then escalate). The underlying problem — 92% temporary regressions, 14% permanently worse, 4% catastrophic self-corruption — is not solved, only bounded.

    


    

      
OP7

      
Demo-to-production gap

      

      
In scope

      
Greenhouse = the production test

      
The Greenhouse is explicitly the production test for OP7. Each orky/ implementation is a working claim that the pillar is achievable — not a demo, not a prototype. FAILURE-MODES.md is the third document in the arc that adversarially tests this claim.
