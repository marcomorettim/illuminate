1 The bias

The unreliability is directional, not random.

Random error would average out; a reader could trust the matrix on balance. This bias does not average out. The panel found the matrix errs the same way across cells, so a careful reader can correct for it rather than distrust everything.

Reviewer 1 named the four mechanisms precisely. Each is a way of making a framework look more finished, more isolated, or more durable than it is.

It understates obsolescence

AutoGen is described as "mature but maintenance mode." That soft-pedals the reality: it is community-managed, bug-and-security-only, no new features, and Microsoft has been reported as "retiring" it. Its successor, the Microsoft Agent Framework, has GA'd. By the June 2026 evaluation date this is close to disqualifying, yet the cell reads as neutral.

It conflates cloud with self-host

On multi-tenancy and security, hosted or commercial capabilities are presented as if they were properties of the open-source artifact. Letta self-host has no per-tenant isolation (one shared password = full server access); LangGraph and the Claude SDK OSS layers give only logical or per-process separation, with real tenancy gated behind commercial planes. This is the recurring error, examined in the next section.

It lags fast-moving SDKs

The OpenAI Agents SDK "single run" descriptor is stale, native subagents and sandboxes shipped around April 2026. And LangGraph's 1.0 GA (Oct 22 2025, stable APIs, ~90M monthly downloads) is omitted while peers' milestones are noted. On a fast-moving field, silence about a milestone is itself a distortion.

It is generous on durability

For checkpoint-based frameworks the matrix reads more resilient than the code is. CrewAI's @persist checkpoints Flow state at method boundaries only and requires the developer to detect failure and manually trigger recovery, there is no automatic crash recovery. "Limited built-in crash durability" undersells that gap.

2 The pattern

The recurring error is open-source read as managed.

One mistake shows up again and again across different rows: a capability that only exists in a vendor's cloud or commercial control plane is written into the cell as though the self-hostable library provided it. The reviewers ask for one consistent distinction the matrix never makes: app-implemented scoping vs enforced isolation, and OSS library vs commercial plane.

Four cells carry it. One is an outright factual error. Severities are the reviewers'.

highLetta multi-tenancy

The matrix credits the self-hosted server with "cloud multi-tenant" isolation. Verified against Letta's own docs and GitHub: the OSS server is single-tenant, protected by one shared LETTA_SERVER_PASSWORD bearer token, no per-user or per-tenant key, no namespace isolation. Native multi-tenancy is an open feature request (issue #2887). It exists only in the hosted Letta Cloud product, not the self-hostable artifact the rest of the row describes. Two of three reviewers rated this HIGH.

highClaude SDK license, a factual error

The cell says "Anthropic Commercial ToS (not OSS)." That is wrong for the SDK code: the claude-agent-sdk-python repo carries an MIT license. Only usage is governed by Commercial ToS, because it calls paid Anthropic model APIs. The matrix conflates code license with service terms and produces an outright incorrect "not OSS" label. (Represented here as an intra-source contradiction, the matrix and Reviewer 3 disagree, not resolved by us re-checking GitHub; the reviewer already did.)

lowLangGraph tenancy

"Tenant isolation only in LangGraph Platform" is directionally right but overstates the OSS side as a tenancy story. In the OSS library, thread_id/namespace are application-level scoping conventions over a shared checkpointer and Store, not an enforced isolation boundary. Two tenants are isolated only if the app author partitions namespaces and DB access correctly; there is no built-in authz. The cell should read "app-implemented scoping, no enforced isolation."

medOpenAI SDK durability & sandbox

"No built-in crash-replay" is correct, but "Sessions persist history; sandbox state" reads as operational durability the SDK does not provide, production crash recovery requires the external (now-GA) Temporal integration, not the SDK itself. Separately, the "sandbox persistent filesystem" is the very new April-2026 SandboxAgent feature, partly platform-bound; listing it flatly overstates its maturity for a production selection.

4 The snapshot

It is a fast-drifting 2026 photograph, and undated.

Reviewer 2 called version-fragility the matrix's main weakness. It captures a Q1–Q2 2026 moment where several headline capabilities are brand-new or in preview, and the panel is explicit that these will drift quickly. An undated matrix invites a reader to treat a preview feature as a stable property.

This is exactly the class of claim the study flags rather than chases: the fragility is stated by the source, so it is carried forward as a caveat, not re-checked.

The capabilities already drifting

The panel names them: OpenAI Sandbox Agents (April 2026, partly hosted), Claude Managed Agents (public beta), Microsoft Agent Framework reaching GA, DSPy GEPA. Each is new or preview. The Managed Agents cell is asked to answer multi-tenancy, durability, and security simultaneously, overstating parity with GA infrastructure like Temporal namespaces.

The one that already flipped

The OpenAI SDK durability story is the clearest drift: the matrix's "no built-in crash-replay" was written before the official OpenAI × Temporal integration moved from public preview (Sept 2025) to GA. The capability didn't just get a footnote, it changed status between snapshot and reading. The fix is procedural: date every maturity cell, beta-flag every preview capability, and never treat preview as GA.

5 Not comparable

Two of the nine don't belong on the same axes.

The most structural caveat isn't about a wrong cell; it's about the frame. Two entries are honestly annotated in the matrix but should not be scored beside the agent frameworks without weighting, because they answer a different question.

Keeping them in the table is fine; ranking them against the others is a category error the reader has to hold in mind.

DSPy is not a runtime

DSPy is an optimizer and programming model: composable Modules and Signatures, optimizer-compiled. Its "state" is compiled prompts and weights (save/load JSON); it has no agent-memory primitive, no sandbox, no durable runtime. Scoring it on durability or multi-tenancy compares a compiler to an operating system.

Temporal is a substrate, not an agent

Temporal is an orchestration substrate with no agent primitives, "agent = your code." Its durability is genuinely best-in-class, but the matrix presents it as nearly free, glossing the real constraint: workflow code must be deterministic, so every nondeterministic LLM/tool call must be pushed into Activities, atop a heavy infra/ops burden (a Cassandra/Postgres/MySQL cluster). Best-in-class durability, but not the LLM-native ease the cell implies.
