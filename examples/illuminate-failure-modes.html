A study of Greenhouse OS · eight failure modes, honestly accounted

Andrej Karpathy catalogued eight systematic ways that LLM coding agents fail, from quietly assuming the wrong requirement to patching a bug they never reproduced. Greenhouse is an operating system that runs exactly these agents, which means the thing most likely to fail is running inside the machine built to keep it honest.

So the real question is simple: does Greenhouse actually contain these failures, or does it merely document them?

It enforces on a descending ladder, and every rung names the gap it can't close.

Four tiers of control, each weaker than the one above it: code-level prevention, prompt-level gates, log-only observation, and one failure left completely open. What makes it trustworthy isn't that it catches everything. It's that it tells you exactly what it doesn't.

1 Prevention

The failures that can do real damage are stopped in hardware.

The failures that actually cause harm all share one trait: to do damage, they have to do something. Write a file, escalate a permission, mutate shared state. Greenhouse treats that as the moment to intervene. Instead of trying to reason an agent out of a bad write, it simply removes the ability to write wherever a write isn't warranted.

This is the load-bearing distinction in the entire system. Model-layer defenses can only ever bound the likelihood of a failure, because they operate on a model that can always be wrong. Deterministic controls bound its consequence, because they operate on capabilities the model simply doesn't hold.

Guarantee strengthDeterministic
Least privilege stops speculative code before it can be written
The input-analyzer and plan-agent are granted read access only. Even if one of them hallucinates a feature nobody asked for, it has no tool with which to build it; the capability isn't in its permission set at all. Writing is reserved entirely for the execute-agent, which takes its scope from the plan-agent's structured output rather than its own judgment. The PhaseGate between planning and execution is where that boundary is enforced: nothing crosses from "proposed" to "written" without passing through it.
Irreversibility is, by itself, a form of control
Session health is tracked by a one-way state machine that has no reset(). Context can move from fertile to compacted to contaminated to depleted, but it can never move back. That sounds like a limitation, and it is, yet it is also the enforcement: because the degradation can't be silently undone, a session that has drifted stays visibly drifted until a human intervenes, rather than quietly papering over its own decay.
correction_register.py

The mechanism, in code. A self-repair attempt only counts against the cap when it carries proof it was verified, so an agent that keeps guessing without ever reproducing the bug is refused and escalated rather than allowed to spin through patches.

2 Gating

Failures that fire before any action meet a gate only as strong as the model's willingness to obey it.

Some failures never touch a tool at all. A silent assumption, a quietly chosen interpretation, an unstated approach: these happen entirely inside the model's reasoning, before a single action is taken. There is nothing physical to intercept, so prevention has to move upstream, into the shape of the request itself.

Greenhouse does this with a typed gate. The input-analyzer must return an explicit list of interpretations, and when more than one plausible reading exists, the orchestrator stops and asks. It works, but it lives at the prompt level, which means it holds only for as long as the model chooses to honor it. This is the softest kind of guarantee the system offers.

Guarantee strengthModel must comply
A typed field forces the model to stop and ask
Since June 2026, the input-analyzer returns a mandatory INTERPRETATIONS field. When it lists more than one reading, the orchestrator emits a DISAMBIGUATION REQUIRED signal and halts before the ideation phase can begin, waiting for the user to confirm a single interpretation. The typed field is the contract; the halt is its enforcement.
Why this rung is softer than it looks
The catch is where the gate lives. It runs at the orchestrator's prompt, not in code, so in principle the model could decide an ambiguous request is unambiguous and sail straight past it. Only the irreversible-action boundary has been hardened into a code-level check; disambiguation itself still rests on cooperation.
orky/gate.py

A typed contract the model is asked to fill. The halt is real, but its reliability rests on the model actually surfacing the ambiguity rather than resolving it silently.

3 Observation

The quiet failures are caught in the record, but only after they have already happened.

Over-abstraction, drive-by refactoring, a vaguely chosen approach: these don't announce themselves. Each one happens in the middle of a legitimate, authorized piece of work, inside what the agent genuinely believes is just doing the task. There is no obvious moment to intercept, because from the inside nothing looks wrong.

So Greenhouse records rather than prevents. Every delegation is logged, the cost of over-abstraction shows up in the token accounting, and the trace is there to review. But all of it lands after the fact. The control can see the failure; it cannot stop it. Closing that gap would take a real-time check that doesn't yet exist.

Guarantee strengthSeen, not stopped
The log records what was decided, never why
The accountability chain captures which approach was chosen and at what point in the run, append-only, so nothing can be quietly rewritten later. What it doesn't capture is the reasoning. Vague Approach is precisely the failure that a rationale would expose, and the absence of that field is the whole of the gap.
The membrane's blind spot is self-generated premises
There is a second blind spot here. The membrane strips injected premises at the perimeter, but a premise the model invents on its own trips no pattern, because it never came from outside. Nothing at runtime catches it; only the human sitting at the escalation boundary does.
accountability_chain.py

The gap made visible: the log faithfully records what was decided and when, but never why. The one field that would make a vague approach auditable is the field that was never written.

4 Unmitigated

One failure sits below every control, and the system's honesty about it is the only safeguard left.

Style drift is the slow migration of a codebase away from its own conventions: naming that shifts, comment density that creeps, formatting that wanders. It fires no injection pattern and triggers no delegation event, so it falls beneath every mechanism the system has. There is, quite simply, no runtime control for it.

What Greenhouse does instead is refuse to pretend otherwise. The documentation states it plainly. That admission is itself the safeguard, because a control that names its own blind spot can be audited and planned around, in a way that a silent, falsely-complete one never can.

Guarantee strengthNone
The same honesty extends to the gaps it did close
The fix for fix-without-reproducing proves a verification was specified, not that it actually ran and passed. And no gap in the whole taxonomy has been shown exploited in practice; the map is architectural, drawn from design rather than from incident data. The document says so, which is exactly the point.
